# 직관생활(jikwan-life) 프로젝트 핸드오프 문서

> **이 문서의 용도**: 새 대화방에서 Claude에게 첫 메시지로 붙여넣어, 지금까지의 맥락·작업 방식·남은 일을 그대로 이어가기 위한 컨텍스트 문서입니다. 새 방에서는 이 문서를 붙여넣고 "이 문서 읽고 이어서 하자"라고 한 뒤 다음 작업을 지정하면 됩니다.

## 1. 프로젝트 개요

- **이름**: 직관생활 (jikwan-life)
- **무엇**: KBO 야구 직관(직접 관람) 기록 PWA(웹앱)
- **타겟 사용자**: 20~30대 여성 KBO 팬
- **핵심 차별점**: 기록 → 예쁜 카드 생성 → 인스타그램 공유 (단순 기록이 아니라 "예쁘게 만들어 자랑하는" 경험)
- **배포**: https://jikwan-life.vercel.app (Vercel, 푸시하면 자동 배포)
- **저장소**: GitHub `tjsdnr03/jikwan-life`
- **참고**: 아내가 운영하는 병행 사업(유꾸마켓, 야구 유니폼 액세서리)이 있으나 이 앱과 별개.

## 3. 작업 방식 (⭐ 가장 중요 — 모델/도구 운용)

### 3-1. 기본 원칙
- 나(사용자)는 **vibe-coding 초보자**다. 초보자 기준으로 친절하고 상세하게 설명할 것.
- **Claude는 코드 파일을 직접 작성하지 않는다.** 대신 **"지시문"**(복사해서 그대로 붙여넣을 수 있는 작업 지시 프롬프트)을 만들어 준다. 내가 그 지시문을 아래 도구들에 붙여넣어 실행한다.
- 토큰 절약을 위해 Claude는 코드 전문을 출력하지 않고 지시문 위주로 움직인다.

### 3-2. 사용하는 도구(모델)와 역할 분담
| 도구 | 담당 영역 | 예시 |
|------|----------|------|
| **Cursor Composer 2.5** | UI·화면·디자인·레이아웃·스타일 (눈에 보이는 것) | 글래스 카드 적용, 색 토큰, 화면 톤 통일, 컴포넌트 배치 |
| **Claude Code (opus)** | 백엔드·데이터·API·DB 스키마·인프라·디버깅·원인 추적 | 인증/세션, KBO 데이터 수집, route.ts, 버그 진단, 측정 |

### 3-3. 지시문 작성 규칙 (Claude가 지킬 것)
- **지시문마다 어느 도구에서 실행할지 명시**한다.
  - UI/디자인/화면 톤/레이아웃 → **Composer 2.5**
  - 데이터/인증/API/스키마/성능/버그 진단 → **Claude Code(opus)**
- 지시문에는 항상 **"수정 범위(어떤 파일만 건드릴지)"**와 **"건드리면 안 되는 것"**을 명시한다.
- 한 번에 하나씩, **한 화면씩** 진행한다. 전체를 한꺼번에 바꾸지 않는다.
- 큰 변경(구조/원인 진단)은 **"수정 전에 진단/분석 결과를 먼저 보고"**하도록 지시한다.

### 3-4. Git / 배포 워크플로
- 커밋·푸시는 **내가 터미널에서 직접** 한다: `git status` → `git add -A` → `git commit -m "..."` → `git push`
- Claude는 작업 묶음이 끝날 때마다 **커밋 메시지 + 명령**을 제공한다.
- 푸시하면 **Vercel 자동 배포**. 폰 테스트는 `jikwan-life.vercel.app`(홈 화면 PWA) 또는 컴퓨터 LAN IP.
- **DB 스키마 변경 시**: Supabase 대시보드 → SQL Editor에서 `ALTER TABLE ...`을 **먼저 실행**한 뒤 코드 푸시. (안 하면 저장이 에러남)

### 3-5. 검증 습관
- AI가 한 작업은 **스크린샷으로 직접 확인**한 뒤 다음 단계로 넘어간다.
- 화면 정적 요소는 PC에서도 확인 가능하지만, **고정 버튼·스켈레톤·PWA 세션 등 동작은 폰에서** 확인한다.

### 3-6. 대화 스타일
- **한국어 격식체**, 상세한 설명 선호.

## 4. 지금까지 완료한 작업

### 4-1. 인증 / 세션 (iOS PWA 재로그인 문제 해결 완료)
- 증상: 홈 화면 PWA로 쓰면 열 때마다 재로그인.
- 원인 추적 결과 핵심 범인은 **랜딩 페이지 라우팅**이었음(세션은 정상이었음).
- 수정: ① 세션 저장을 쿠키 → **localStorage**(`@supabase/supabase-js`의 `createClient`, `persistSession:true`, `autoRefreshToken:true`, 싱글톤화) ② 부팅 인증 가드를 `getUser()` → **`getSession()`**(콜드 스타트 네트워크 미준비 시 튕김 방지) ③ 랜딩(`/`)에 **"세션 있으면 `/home`으로 `router.replace`"** 분기 추가.
- 마이 탭에 로그아웃 버튼 있음(`signOut` 후 랜딩/로그인으로).

### 4-2. 전체 글래스 UI 리디자인 (완료)
- 적용 완료 화면: 홈(`/home`), 기록 목록(`/record`), 통계(`/stats`), 달력(`/calendar`), 마이(`/my`), 랜딩(`/`), 로그인(`/login`), 그리고 하위 페이지(기록 작성 `/record/new`, 수정 `/record/[id]/edit`, 상세 `/record/[id]`, 카드 만들기 `/card`·`/card/[id]`·`/card/preview`의 셸).

### 4-3. 색 톤 (현재 확정 상태)
- 배경: **퓨어 화이트 `#ffffff`** (이전 웜 오프화이트 `#f6f4f0`에서 변경)
- 카드 표면: **쿨 그레이 `#f1f3f6`** (`.glass-card`)
- accent: 블루 (모든 화면 공통 토큰)
- 텍스트: `text-text-primary/secondary/tertiary` 토큰
- 글래스 카드의 흰색 보더 하이라이트는 제거함(쿨 그레이 위에서 흰 선이 도드라져서)
- ⚠️ 이 색들은 **globals.css의 전역 토큰**이라 한 번 바꾸면 모든 화면에 반영됨
- 글래스의 "유리 질감"은 약함(단색 배경이라 blur가 비칠 대상이 없음). 현재는 "클린 미니멀" 방향으로 수용한 상태. 웹 PWA의 글래스 한계로 합의됨.

### 4-4. 하단 탭바
- 메인 5탭 + **하위 페이지(기록 상세/작성/수정, 카드 만들기)에도 하단 글래스 탭바 추가**(BottomNav, `variant="glass"`). 활성 표시는 "기록".

### 4-5. 기록 목록 디테일
- 사진 썸네일: 우측 고정 크기 정사각 정렬, 여러 장이면 장수 배지, 대표 1장만(전체는 상세에서)
- 승/패/무 배지: 사진 유무와 무관하게 **우측 상단 고정**
- 결과 배지 체계: 승 = `accent-bg`/`accent`, 패·무 = 회색 계열

### 4-6. 경기 데이터 (진행 중)
- **데이터 소스 조사 완료** (아래 8번 참고)
- **경기 시작 시간**: `kbo_games`에 `game_datetime`(TIMESTAMPTZ), `time_tbd`(BOOLEAN) 컬럼 추가 + `route.ts`에서 저장. (Supabase SQL로 컬럼 추가 완료, 푸시 완료) → 현재 **API 응답에 싣기 + 홈 카드에 표시** 작업 진행 중.
- **라이브 상황판 데이터 조사 완료**: 이닝·아웃·볼스트라이크·주자(다이아몬드)가 `relay` 엔드포인트의 `currentGameState`에 있고, 선수명 없이 상황값만 분리 가능. 단 투구 단위(~12초)로 바뀌어 폴링 필요 → 부담 + "관전 앱(빅이닝)" 영역이라 **풀스펙은 보류 검토 중**. 가벼운 스냅샷(스코어+이닝) 쪽으로 기울어진 상태.

## 6. 남은 할 일 (백로그 / 우선순위)

1. **경기 시작 시간 표시 마무리** — 홈 카드(진행 중) + 달력에도 적용
2. **종료 경기 이닝별 라인스코어(스코어보드)** — `record` 엔드포인트, 경기당 1회 수집(부담 없음), `inning_scores` 컬럼 채우기. 기록 상세/카드에 표시
3. **라이브 스냅샷 범위 결정** — 가벼운 쪽(스코어+이닝, 진입 시 1회) 우선. 주자 다이아몬드 풀스펙은 넣더라도 "진행 중 경기 상세"에 한정
4. **통계 확장** — 다른 기록 앱들이 쓰는 재미있는 통계 추가 (데이터 조사 결과 기반)
5. **달력 표시 개선** — 홈/원정 테두리 동그라미 제거 → 직관한 경기의 승/패/무를 각 캐릭터 뒤 **배경색 큰 원**으로 표시 (동그라미가 너무 많아 헷갈린다는 피드백)
6. **색상 / 응원팀 색 테마 + 전체 색 조화** — 마이 탭에서 팀 색 선택. 배경 톤은 웜 vs 쿨(퓨어 화이트+쿨 그레이) 비교했고 현재 쿨 적용. 응원팀 색을 accent로 바꾸는 시스템 설계 필요. globals.css 전역 토큰이라 신중히
7. **화면 전환 속도 개선** — 웹앱에서 화면 전환 딜레이/끊김. 원인 진단 필요(프리페치/데이터 페칭/캐싱 의심). Claude Code 영역
8. **캐릭터 작업** — 캐릭터 디자인 + 캐릭터가 들어갈 자리 편집. 일러스트 필요한 별도 트랙, 가장 무거움. (방향: 단일 베이스 고양이 + 팀별 동물 모티프 액세서리, 빅이닝 방식 참고)

## 7. 불변 제약 (반드시 지킬 것)

1. **KBO 라이선스**: 실제 구단 로고·마스코트·**선수 이름** 사용 금지. 자체 **고양이 캐릭터 + 팀 색**만 사용.
2. **데이터도 선수 개인정보(선수명·라인업·투수/타자 박스스코어) 금지** — 경기·팀 단위 데이터만. (relay의 주자 점유는 `!= "0"` 불리언으로만, pitcher/batter 코드는 무시)
3. **StoryCard / CardEditor의 카드 미리보기**는 html2canvas-pro 추출용 인라인 스타일 — **건드리면 카드 저장/공유가 깨짐**. 카드 만들기 화면은 "셸(배경·헤더·컨트롤)"만 새 톤 적용, 가운데 카드 미리보기는 손대지 않음.
4. **Claude는 코드를 직접 쓰지 않고 지시문만** 만든다. 지시문마다 실행 도구(Composer / Claude Code)와 수정 범위를 명시.
5. **DB 스키마 변경 시** Supabase SQL Editor에서 `ALTER TABLE` 먼저 실행 후 푸시.
6. 회사 업무용 Excel은 xlwings 사용(이 앱과 무관하지만 참고).

## 8. 데이터 소스 메모 (KBO)

- 출처: **네이버 스포츠 비공식 API** (`api-gw.sports.naver.com`). 호출 시 User-Agent + Referer 헤더 필요. 비공식이라 스키마가 예고 없이 바뀔 수 있어 **방어적 파싱** 필요.
- 캐시: Supabase `kbo_games` 테이블에 upsert.
- 엔드포인트:
  - `/schedule/games` — 일정/결과 (현재 사용 중). 필드: gameId, gameDate, **gameDateTime**(KST, 오프셋 없음 → +09:00 붙여 변환), timeTbd, stadium, home/awayTeamName, scores, winner, statusCode, **statusInfo**("9회말") 등. ⚠️ 선수명 필드(starterName 등)는 제외.
  - `/schedule/games/{gameId}/record` — 종료 경기: **라인스코어**(scoreBoard.inn, rheb), 팀 시즌 성적(standings), 그 경기 팀 집계(todayKeyStats). ⚠️ etcRecords·boxscore는 선수명 포함 → 선별.
  - `/schedule/games/{gameId}/preview` — 예정 경기: 시즌 상대전적(seasonVsResult), 최근 5경기 폼(previousGames), 순위(standings). ⚠️ 라인업/선발 제외.
  - `/schedule/games/{gameId}/relay` — 라이브: `textRelayData.currentGameState`에 out/ball/strike/base1·2·3/score, `textRelayData.inn`·`homeOrAway`. ⚠️ 선수 코드·중계 문장·라인업 제외. cache 없이 폴링 설계(~12초). 진행 중(STARTED)일 때만 폴링하도록 상태 게이팅 필요.
- 비용 메모: `record`/`preview`/`relay`는 경기당 1콜 추가 → 캐싱 필수. 목록은 `schedule`로 저비용, 상세 진입 시에만 무거운 엔드포인트 호출하는 2단 구성 권장.

## 9. 새 대화방에서 시작하는 법

1. 이 문서 전체를 새 대화방 첫 메시지로 붙여넣는다.
2. "이 문서 읽고 이어서 하자"라고 말한 뒤, 백로그(6번) 중 다음에 할 작업을 지정한다.
3. Claude는 맥락을 잡고, 해당 작업의 지시문(실행 도구 명시)을 만들어 준다.
